# 视频语义校验层 · 前端修改报告

> 本报告记录基于 `新任务.md` 的"检测—语义双通道视频证据理解机制"前端改造工作。
> 后端三个核心接口（`video-semantic-check` / `fuse-video-evidence` / `fused-evidence`）及融合规则 `fuse_video_evidence` 已由后端团队实现，本次工作聚焦**前端对接与展示改造**。

---

## 一、修改背景

### 原架构
```
视频上传 → 神经网络视频检测模型 → Dify 责任推理 → 报告
```

### 新架构
```
视频上传 → 关键帧抽取 → 原视频检测模型 → 千问视频语义校验模型
       → 证据融合与冲突门控 → 结构化证据包 → Dify 责任推理 → 报告
```

### 职责边界
- **检测模型**：负责"看到了什么"（车辆目标、关键帧、事故时间点）
- **千问语义校验**：负责"这段视频在发生什么"（视角识别、自车隐式参与、车辆行为语义）
- **融合层**：负责"这些证据能不能进入责任推理"（冲突门控、人工复核判定）
- **Dify**：基于融合证据包生成责任说明，不直接定责

## 三、详细修改内容

### 3.1 P0 · API 层接口对接

**文件**：`src/api/index.js`

**改动**：在 `CasesAPI` 对象中新增 3 个方法，对接后端已实现的 3 个接口。

```javascript
// 1. 获取融合证据包（前端展示用，最常用）
async getFusedEvidence(caseId) {
  return request('GET', `/api/cases/${caseId}/fused-evidence`)
},

// 2. 触发千问视频语义校验（通常由后端自动串联，前端仅在手动重试时调用）
async videoSemanticCheck(caseId, data) {
  return request('POST', `/api/cases/${caseId}/video-semantic-check`, data)
},

// 3. 触发证据融合（通常由后端流程自动调用）
async fuseVideoEvidence(caseId, data) {
  return request('POST', `/api/cases/${caseId}/fuse-video-evidence`, data)
}
```

**要点**：
- 前端**不直接调用千问 API**，只调自己后端，避免 API Key 暴露
- 前端主要使用 `getFusedEvidence` 拉取已融合的证据包进行展示
- `videoSemanticCheck` 和 `fuseVideoEvidence` 仅在需要手动重试时调用

## 四、技术要点

### 4.1 真实接口对接
所有数据来自后端 `GET /api/cases/{case_id}/fused-evidence`，前端不调用千问 API，不硬编码 API Key，不使用任何假数据或预设接口。

### 4.2 空值兜底
后端在无数据时返回空对象 `{}`，前端做两层过滤：
```javascript
const packet = result.data?.fused_evidence_packet
fusedEvidence.value = (packet && Object.keys(packet).length > 0) ? packet : null
```
所有字段访问使用可选链 `?.` 并提供 `—` 默认值，避免空值导致渲染异常。

### 4.3 中文映射表
所有英文枚举值通过映射表转为中文，不裸露英文：

| 英文字段 | 中文展示 |
|---------|----------|
| `dashcam_ego_view` | 行车记录仪视角 |
| `lane_change` | 变道 |
| `rear_end` | 追尾 |
| `lane_change_or_cut_in` | 变道/切入 |
| `needs_manual_review` | 需人工复核 |
| `manual_review_required` | 需人工复核 |
| `not_supported` | 不支持 |
| ... | ... |

每个页面独立维护映射函数，未找到时返回原值或 `—`，避免新枚举值导致展示异常。

### 4.4 冲突可视化
跨 4 个页面保持一致的冲突可视化：
- **冲突状态**：橙色高亮（`#FF9500`）
- **人工复核**：红色标记（`#FF3B30`）
- **通过状态**：绿色标记（`#34C759`）

### 4.5 并行数据加载
所有页面的数据加载均使用 `Promise.all` / `Promise.allSettled` 并行请求，融合证据包加载不阻塞原有数据加载：
- EvidenceChain：7 个并行请求
- RuleGraph：5 个并行请求
- ReportDetail：2 个并行请求
- IntelligentAnalysis：3 个并行请求
