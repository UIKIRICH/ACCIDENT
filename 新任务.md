原视频检测模型继续保留，千问视觉/视频多模态模型作为“视频语义校验层”，两者融合后再交给 Dify 做责任推理。

现在不是只加一个模型，而是要把整条视频证据链改顺。下面我按“必须改 / 建议改 / 展示改 / 测试改”给你列一套具体措施。

一、整体架构怎么改

原来大概是：

视频上传
→ 神经网络视频检测模型
→ 输出事故类型、车辆数、关键帧、置信度
→ Dify 责任推理
→ 责任建议 / 规则依据 / 报告

现在改成：

视频上传
→ 关键帧抽取
→ 原视频检测模型
→ 千问视频语义校验模型
→ 证据融合与冲突门控
→ 结构化证据包
→ Dify 责任推理
→ 规则依据 / 人工复核 / 报告导出

一句话：

原模型负责“看到了什么”，千问负责“这段视频在发生什么”，融合层负责“这些证据能不能进入责任推理”。

二、第一处必须改：车辆数量字段

现在的：

车辆数量 vehicle_count = 1

必须废掉，不要再直接作为“涉事车辆数量”。

改成三个字段：

{
  "visible_external_vehicle_count": 1,
  "ego_vehicle_present": true,
  "estimated_involved_vehicle_count": 2
}

页面展示改成：

画面可见外部车辆数：1
视频视角：行车记录仪视角
隐式自车：存在
估计涉事车辆数：2

解释：

行车记录仪视角下，自车通常不会以目标框形式出现在画面中，因此画面检测车辆数不等于事故涉事车辆总数。

这一步必须做，不然后面还会出现“一辆车怎么追尾/变道”的尴尬。

三、第二处必须改：事故类型字段

现在的：

事故类型 accident_type = rear_end
事故类型置信度 type_confidence = 0.333

不能直接展示成“事故类型：追尾”。

改成：

{
  "candidate_accident_type_from_detector": "rear_end",
  "detector_type_confidence": 0.333,
  "semantic_accident_type_from_qwen": "possible_lane_change_or_cut_in",
  "accepted_accident_type": "unknown",
  "type_decision_status": "needs_manual_review"
}

页面展示：

检测模型候选类型：追尾事故，置信度 0.333
语义校验结果：疑似前车变道 / 切入 / 侧向接触
最终事故类型：暂不直接认定
系统动作：进入人工复核

这样就不会出现“明明像前车变道，系统却硬说追尾”的问题。

四、第三处必须改：追尾匹配度字段

现在的：

追尾匹配度 rear_end_match = 0.000

这个字段不能裸露展示。

改成：

{

    "score": 0.000,
    "status": "not_supported",
    "reason": "未检测到明确前后车追尾关系，且语义校验更倾向于变道/切入关系"

追尾特征支持：未通过
说明：当前证据未能支持标准追尾关系，检测模型的追尾候选不被直接采纳。

这个字段以后只作为“事故类型支持证据”，不是单独给评委看的神秘参数。

五、第四处必须改：证据一致性字段

现在的：

必须解释来源。

改成：

{
  "keyframe_video_consistency": {
    "score": 0.469,
    "level": "low",
    "comparison_target": "keyframes_vs_video_detection_and_semantic_check",

七、千问模型的 Prompt 怎么写

你们给千问的 prompt 必须严格，不要让它自由发挥。

可以这样写：

你是交通事故视频语义校验模块。请只基于给定关键帧、短视频片段和检测模型输出进行分析，不得编造画面外事实，不得直接进行责任认定。

你的任务不是判责，而是输出视频语义观察结果，用于后续规则推理和人工复核。

请重点判断：
1. 视频是否为行车记录仪视角；
2. 自车是否应作为隐式事故参与方；
3. 画面中可见外部车辆数量；
4. 外部车辆是否存在变道、切入、急停、侧向靠近等行为；
5. 检测模型给出的事故类型候选是否被画面证据支持；
6. 当前证据是否足以进入责任推理；
7. 缺少哪些关键证据。

请严格输出 JSON，不要输出多余文字。

输出格式：

{
  "camera_view": "dashcam_ego_view / roadside_view / surveillance_view / unknown",
  "ego_vehicle_present": true,
  "visible_external_vehicle_count": 1,
  "estimated_involved_vehicle_count": 2,
  "external_vehicle_behavior": "lane_change / cut_in / braking / straight / unknown",
  "ego_external_relation": "external_vehicle_moves_into_ego_path / ego_rear_ends_front_vehicle / side_collision / unknown",
  "semantic_accident_type_candidate": "lane_change_or_cut_in / rear_end / side_collision / unknown",
  "semantic_confidence": 0.0,
  "supporting_observations": [],
  "conflict_with_detector": true,
  "conflict_reasons": [],
  "missing_evidence": [],
  "should_enter_liability_reasoning": false,
  "manual_review_required": true
}
八、融合规则怎么做

第一版不要搞复杂权重。先用规则门控。

规则 1：千问识别为行车记录仪视角
如果 camera_view = dashcam_ego_view：
ego_vehicle_present = true
estimated_involved_vehicle_count = visible_external_vehicle_count + 1
规则 2：检测模型置信度低
如果 detector_type_confidence < 0.5：
不得直接采纳检测模型事故类型
规则 3：追尾支持为 0
如果 candidate_type = rear_end 且 rear_end_support < 0.3：
追尾候选不进入最终事故类型
规则 4：两个模型冲突
如果 detector_candidate_type 和 qwen_semantic_type 不一致：
conflict_detected = true
manual_review_required = true
规则 5：千问只作为语义校验，不直接定责
千问不能输出最终责任；
千问只能输出事实观察、证据缺失和人工复核建议。
规则 6：证据不足时强制人工复核
如果 missing_evidence 非空且涉及碰撞部位、双方车辆关系、事故后位置：
manual_review_required = true
accepted_accident_type = unknown

十五、最关键的结论

你们这次不是“为了炫技多接一个模型”，而是在补真实系统的短板：

单一神经网络视频检测模型能提取画面目标，但难以理解行车记录仪视角、自车隐式参与、车辆变道切入等复杂语义。
因此系统增加千问视频语义校验层，用于对检测结果进行二次解释、冲突识别和证据补全。
该层不直接判责，只生成结构化事实和证据缺失项，最终仍由规则依据与人工复核完成责任建议。

这个逻辑非常稳。

四、PPT/答辩需要改的部分

这部分不是代码，但很重要。

不要说：

我们用千问来判责。

要说：

系统引入视频语义校验层，用于弥补单一检测模型在行车记录仪视角、自车隐式参与和复杂车辆行为语义理解上的不足。该层不直接输出责任结论，只生成结构化观察和证据冲突提示，最终责任建议仍由规则依据与人工复核形成。

最适合你们项目的方案应该是：

YOLO/原模型做主检测，千问做语义校验与纠偏，后端融合层做最终门控。

也就是说，千问不是“再独立判一次责任”，而是帮你们判断：

这个视频是什么视角？
是不是行车记录仪？
自车是不是隐式参与？
YOLO 的车辆数量是否需要补偿？
YOLO 的事故类型候选是否明显不合理？
有没有证据冲突或缺失？

然后由后端融合层决定：

能不能进入 Dify 责任说明？
要不要人工复核？
要不要提示证据不足？
最推荐方案：主模型 + 语义校验，不是双模型平权判断

你们原本想法其实是对的：

千问主要解决车载视频、行车记录仪、自车不可见、复杂语义这类 YOLO 不擅长的问题。

所以不要设计成：

YOLO 判一次
千问也判一次
Dify 再自己选谁对

这个会乱，也会像黑箱。

应该设计成：

YOLO：输出检测事实和候选类型
千问：检查这个视频语义和 YOLO 结果是否合理
融合层：根据规则决定是否采纳、修正或转人工
Dify：只负责生成解释和报告

这才是最稳的。

三种方案对比
方案 A：只用 YOLO
YOLO → Dify → 责任建议

优点是简单。
缺点是车载视频很容易翻车，比如“画面只有 1 辆车，但实际自车参与”。

不建议继续只用这个。

方案 B：YOLO 和千问都判断，Dify 自己综合
YOLO 输出一份结论
千问输出一份结论
Dify 自己综合

这个表面看智能，但风险最大。

因为 Dify 可能会被 YOLO 的 rear_end 带偏，也可能会被千问的自然语言带偏，最后变成：

大模型自己在判责。

答辩时也不好解释：
“那到底谁是裁判？YOLO？千问？Dify？”

不推荐。

方案 C：YOLO 主检测 + 千问语义校验 + 融合层裁判
YOLO 检测事实
↓
千问语义校验
↓
后端融合门控
↓
Dify 生成责任说明/人工复核说明

这个最好。

因为责任边界清楚：

YOLO：看见什么
千问：理解语义、检查矛盾
融合层：决定证据是否可用
Dify：把融合结果说清楚
人工：最终复核确认

这就是你们应该选的。

所以千问到底输出什么？

千问不要输出：

A 车主责
B 车次责
最终责任为……

千问应该输出：

视角类型：行车记录仪 / 监控视角
自车是否隐式参与：是/否
可见外部车辆数：1/2/3
估计涉事车辆数：2/3
外部车辆行为：变道 / 切入 / 急停 / 侧滑
语义事故类型候选：追尾 / 侧向碰撞 / 变道碰撞
与检测模型是否冲突：是/否
缺失证据：碰撞部位、事故后位置、车损图片等
建议动作：进入规则推理 / 人工复核 / 补充证据

也就是说，千问做的是语义校验，不是责任认定。

那 Dify 需要几个输入？

从工程上可以给 Dify 一个输入包，但里面有层级。

最稳的是：

主输入：fusion_result + fused_evidence_packet
辅助材料：YOLO 原始输出 + 千问原始输出

Dify 的优先级必须写死：

第一优先级：fusion_result
第二优先级：fused_evidence_packet
第三优先级：YOLO / 千问原始输出，仅用于解释溯源

也就是说，Dify 可以看到两个原始结果，但不能自己绕过融合层重新判。
